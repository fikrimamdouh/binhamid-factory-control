# المرحلة الثانية: تفعيل نظام مصنع بن حامد الكامل

## 1. النشر

1. ادمج أو انشر الفرع المعتمد بعد انتهاء مراجعة الكود.
2. تأكد أن Vercel يبني أحدث Commit وأن عدد Functions هو 6.
3. ثبّت Node على `24.x` كما هو محدد في `package.json`.
4. أضف متغيرات `.env.example` إلى بيئة Production فقط.
5. نفّذ Redeploy بعد حفظ المتغيرات.

## 2. قاعدة البيانات

شغّل الملفات التالية في Supabase SQL Editor بالترتيب:

1. `001_initial_schema.sql`
2. `002_driver_attendance_and_roles.sql`
3. `003_enterprise_operations_and_conversations.sql`
4. `004_operational_projection_backfill.sql`
5. `005_enterprise_runtime_completion.sql`
6. `006_runtime_replay_and_integrity.sql`

الملفات مصممة لتكون قابلة لإعادة التشغيل دون مسح البيانات.

بعد التنفيذ افتح من جلسة الإدارة:

```text
/api/system/database-readiness
```

يجب أن تكون النتيجة:

```text
ready=true
schemaVersion=6
```

## 3. Storage

أنشئ Bucket خاصًا باسم القيمة الموجودة في:

```text
SUPABASE_STORAGE_BUCKET
```

القيمة الافتراضية:

```text
factory-documents
```

لا تجعل الـBucket عامًا. تنزيل الملفات يتم من الخادم بعد التحقق من صلاحية الإدارة.

## 4. Telegram

1. استبدل رمز البوت إذا سبق نشره في محادثة أو ملف.
2. أضف `TELEGRAM_BOT_TOKEN` و`TELEGRAM_WEBHOOK_SECRET` و`TELEGRAM_OWNER_ID` إلى Vercel.
3. افتح مركز الاتصال ثم إعداد الخادم.
4. اضغط تسجيل Webhook.
5. افتح:

```text
/api/telegram/status
```

يجب أن يظهر أن الـWebhook يشير إلى:

```text
/api/telegram/webhook-v3
```

## 5. المستخدمون والمواقع

1. يسجل كل موظف `/start` أو `/whoami`.
2. اعتمد الحساب من مركز الاتصال وحدد الدور.
3. افتح إدارة الحضور والسائقين.
4. سجل مواقع العمل وإحداثياتها ونطاق الحضور.
5. اربط كل حساب Telegram بسجل الموظف والموقع والوردية.
6. اربط السائق بالمركبة أو الأصل.
7. اختبر الحضور والانصراف من الهاتف باستخدام GPS الحالي.

## 6. الجدولة والتنبيهات

في Vercel أضف:

```text
CRON_SECRET
```

في GitHub Repository Secrets أضف:

```text
BINHAMID_CRON_SECRET
BINHAMID_CRON_URL
```

`BINHAMID_CRON_SECRET` يساوي نفس قيمة `CRON_SECRET`.

`BINHAMID_CRON_URL` يكون مثل:

```text
https://YOUR_DOMAIN/api/cron/manager-brief
```

Workflow `Operational Scheduler` يعمل كل ساعة ويعالج:

- صندوق التنبيهات.
- إعادة محاولة التنبيهات الفاشلة.
- تنبيهات الإدارة المهمة.
- تذكير الموظفين الذين لم يسجلوا التقرير اليومي.

## 7. الخدمات الخارجية

### OpenAI

يُستخدم لفهم الصوت، تحويل الصوت إلى نص، الرد الطبيعي، وفهم المستندات.

### Google Places

يُستخدم للبحث عن الموردين وبيانات الاتصال العامة.

### GPS

يمكن ربط Traccar أو مزود الأسطول الفعلي باستخدام متغيرات `GPS_*`.

### PDF

اختياري. عند عدم تفعيله يرسل البوت مستند HTML قابلًا للطباعة. عند تفعيله يستخدم `PDF_API_URL` و`PDF_API_KEY`.

### رابط الإنتاج

أضف `PUBLIC_APP_URL` حتى تحمل المستندات رابط تحقق صحيحًا.

## 8. اختبار القبول حسب الدور

اختبر حسابًا منفصلًا لكل دور:

- مدير النظام.
- مدير المصنع.
- المحاسب.
- الميكانيكي.
- مبيعات البلوك.
- مبيعات الخرسانة.
- المحصل.
- السائق.
- الموظف.
- مسؤول المخزن.
- مسؤول الديزل.
- الموارد البشرية.
- المشتريات.
- الجودة.

لكل دور تحقق من:

1. ظهور القائمة المناسبة فقط.
2. إنشاء عملية بالأزرار ومراجعتها قبل الحفظ.
3. ظهور العملية فورًا في مركز العمليات.
4. تحديث الحالة وإرسال إشعار لصاحب العملية.
5. تطبيق الاعتماد أو الرفض عند الحاجة.
6. ظهور العملية في مركز التقارير.
7. منع الوصول إلى الوظائف غير المسموحة.

## 9. دورات القبول الكاملة

### المبيعات

```text
إنشاء أمر بيع → اعتماد → تجهيز → توريد → تسليم → تحصيل → تقرير
```

### الورشة

```text
بلاغ → فحص → طلب قطع → عروض أسعار → اعتماد → إصلاح → اختبار → إغلاق
```

### المشتريات والمخزون

```text
طلب شراء → اعتماد → عروض → استلام → إضافة مخزون → صرف مرتبط → جرد
```

### الموظفون

```text
اعتماد الحساب → ربط الموقع → حضور GPS → تقرير يومي → مهمة → إغلاق
```

### السائق

```text
حضور → بداية رحلة → عداد → تحميل → وصول → تسليم → ديزل → إنهاء رحلة
```

## 10. التحقق النهائي

- `/api/system/status` يعرض `directOperationsSchema=6`.
- `/api/system/database-readiness` يعرض `ready=true`.
- `/api/telegram/status` يعرض Webhook v3.
- مركز المحادثات يعرض الرسائل الواردة والصادرة.
- مركز العمليات يسمح بإنشاء مهمة وتحديث الحالات والاعتماد.
- مركز التقارير يعرض كل الأقسام حسب الفترة.
- صفحة `verify-document.html` تتحقق من رمز المستند.
- الحضور خارج النطاق يُسجل كمخالفة ولا يُعتمد حضورًا صحيحًا.
- الديزل والعدادات والرحلات تظهر في تقرير الأسطول.
- لا يزيد عدد Vercel Functions عن 12؛ العدد المتوقع 6.
