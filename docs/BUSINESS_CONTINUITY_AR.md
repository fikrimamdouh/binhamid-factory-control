# خطة استمرارية وتشغيل طوارئ مصنع بن حامد

## الهدف

استمرار تسجيل العمليات ومنع فقد البيانات عند تعطل الإنترنت أو Vercel أو Supabase أو Telegram، مع استعادة الخدمة بطريقة قابلة للإثبات.

## مستويات الحادث

| المستوى | الوصف | الإجراء |
|---|---|---|
| 1 | بطء أو فشل مؤقت لا يمنع العمل | إعادة المحاولة وتسجيل التنبيه |
| 2 | توقف خدمة واحدة مع بقاء الحفظ المحلي | تشغيل محلي مؤقت ومنع الاعتماد النهائي |
| 3 | توقف الحفظ السحابي أو تعارض Revision | وقف الاعتماد، حفظ نسخة محلية، فتح حادث |
| 4 | اشتباه فقد أو تلف بيانات | وقف الكتابة، إنشاء Backup، مراجعة الأدلة |
| 5 | اختراق مفتاح أو حساب | تدوير السر فورًا، إبطال الجلسات، مراجعة Audit |

## التشغيل عند انقطاع الإنترنت

1. يستمر إدخال البيانات في النسخة المحلية فقط.
2. يمنع اعتماد التقرير اليومي أو إقفال الفترة أثناء غياب السحابة.
3. يظهر للمستخدم أن النسخة محلية وغير متزامنة.
4. بعد عودة الاتصال، تتم قراءة Revision السحابية قبل الحفظ.
5. عند التعارض لا تُستبدل النسخة الأحدث تلقائيًا؛ تُعرض المقارنة ويختار المسؤول.

## توقف Vercel

- لا تغيّر قاعدة Supabase.
- احتفظ بالبيانات المحلية والملفات الأصلية.
- راجع GitHub Actions وحالة آخر Deployment.
- يمنع إعادة نشر متكرر عند وجود حد نشر؛ يستخدم البناء المحلي والفحوص لتحديد سلامة الكود.
- بعد عودة النشر، شغّل Production Readiness قبل الاعتماد.

## توقف Supabase

- توقف جميع الاعتمادات والتعديلات السحابية.
- يستمر التسجيل المحلي مع وسم «غير متزامن».
- لا تنفذ Migration أو Restore أثناء عدم استقرار الخدمة.
- بعد عودة الخدمة: فحص الاتصال، Schema، Revision، ثم مزامنة محكومة.

## توقف Telegram

- البرنامج الأساسي يستمر.
- الملفات ترسل لاحقًا بعد عودة البوت.
- تراجع `notification_outbox` للرسائل الفاشلة وDead Letter.
- لا تعاد الرسالة يدويًا أكثر من مرة دون مراجعة dedupe key.

## النسخ الاحتياطي

- نسخة مشفرة يومية عبر GitHub Actions.
- نسخة مشفرة قبل وبعد كل Migration.
- كل Artifact يحتوي `.enc` وmanifest وchecksum ونتيجة تشغيل.
- metadata تسجل داخل `backup_runs`، ولا يسجل مفتاح التشفير أو رابط الاتصال.
- لا تعتبر النسخة صالحة قبل تحقق الحجم والـchecksum.

## اختبار الاستعادة

- يتم أسبوعيًا وبعد Migrations الجوهرية.
- الاستعادة تكون إلى PostgreSQL 17 معزول غير إنتاجي.
- تتم مطابقة Schema وعدد الصفوف الأساسية والكائنات المطلوبة.
- يحذف ملف SQL المفكوك فور انتهاء الاختبار.
- تسجل النتيجة داخل `restore_test_runs`.
- ممنوع Restore على Production إلا بقرار حادث رسمي وخطة معتمدة.

## استعادة Production عند كارثة مؤكدة

1. إيقاف الكتابة والنشر.
2. تحديد آخر Backup verified قبل الحادث.
3. التحقق من checksum والـmanifest.
4. تكرار Restore Drill على قاعدة معزولة.
5. مقارنة أعداد العملاء والمبيعات والتحصيلات والصيانة.
6. الحصول على موافقة الإدارة والمالية ومدير النظام.
7. إنشاء قاعدة Production بديلة أو نقطة استعادة منفصلة؛ لا تكتب فوق القاعدة الحالية مباشرة.
8. تحويل التطبيق بعد قبول النتيجة.
9. حفظ القاعدة القديمة للطب الشرعي وعدم حذفها.
10. إصدار تقرير حادث ومحضر قرار.

## تدوير الأسرار

- كل 90 يومًا أو فور الاشتباه.
- لا تخزن القيم داخل GitHub Issues أو الكود أو قاعدة البيانات.
- يسجل اسم السر وتاريخ التدوير فقط في `token_rotation_registry`.
- بعد تدوير Telegram يعاد تسجيل Webhook.
- بعد تدوير قاعدة البيانات يعاد اختبار Backup وRestore Drill.

## الأدلة المطلوبة بعد كل حادث

- رقم Workflow أو Deployment.
- Commit المستخدم.
- وقت بداية ونهاية الحادث.
- الأنظمة المتأثرة.
- آخر Backup verified.
- نتيجة Restore Drill.
- الإجراءات والأشخاص المنفذون.
- الصفوف أو المستندات المتأثرة إن وجدت.
- سبب الجذر والإجراء الوقائي.
