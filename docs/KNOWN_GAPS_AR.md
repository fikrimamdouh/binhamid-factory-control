# سجل الفجوات المعروفة وموانع التسليم

> الحالة المرجعية: `main` عند `917c74f4be825880b865a5a6257b66f9003b3063`.

## طريقة التصنيف

- **P0:** خطر مالي، فقد/تكرار بيانات، تجاوز صلاحية، أو مانع مباشر للتسليم.
- **P1:** نقص تشغيلي أو رقابي جوهري يجب إغلاقه قبل التشغيل الكامل.
- **P2:** دين فني وتحسين مطلوب بعد تثبيت السلوك.

لا تغلق الفجوة بمجرد دمج كود؛ تغلق فقط بعد نجاح اختبارها وقبول دليل التشغيل.

## P0 — موانع التسليم

### GAP-P0-01 — لا يوجد محرك عمليات موحد عام

**الحالة:** مفتوح.

**الدليل:** التقرير اليومي يستخدم خدمة/RPC موحدة، بينما المبيعات والمهام والاعتمادات تستخدم منطقًا مستقلًا داخل Bot/Route files.

**الخطر:** اختلاف قواعد التحقق والحالات والصلاحيات بين الموقع وتيليجرام.

**الإغلاق المطلوب:**

- `executeOperation()`.
- `approveOperation()`.
- `rejectOperation()`.
- `postOperation()`.
- `reverseOperation()`.
- `getOperationStatus()`.
- Adapter رفيع للموقع وتيليجرام.

**PR المستهدف:** PR-02.

### GAP-P0-02 — الفوترة من أمر البيع لا تنتج فاتورة وقيدًا

**الحالة:** مفتوح ومثبت في PR #77 و`bot-sales.js`.

**الدليل:** تغيير الحالة إلى `invoiced` يرسل تنبيهًا بأن القيد يأتي من التقرير اليومي أو شاشة اعتماد أخرى.

**الخطر:** أمر مفوتر شكليًا بلا مستند مالي أو قيد، أو إنشاء قيد لاحق غير مربوط.

**الإغلاق المطلوب:**

- فاتورة فعلية برقم فريد.
- ربط بأمر البيع والعميل.
- قيد واحد فقط.
- تحديث المديونية.
- عكس رسمي عند الإلغاء بعد الترحيل.
- Unique constraints وIdempotency.

**PR المستهدف:** PR-03.

### GAP-P0-03 — استخدام `audit_log` بدل سجل العملية الحقيقي

**الحالة:** مفتوح في مسارات مثبتة.

**أمثلة:**

- `createManagementTask()` يكتب حدثًا في `audit_log`، بينما التقارير تقرأ `operational_tasks`.
- `setOperationStatus()` يسجل الحالة في `audit_log` دون تحديث مثبت لـ`operational_records`.

**الخطر:** اختلاف الحقيقة بين الصفحة، التقرير، والبوت.

**الإغلاق المطلوب:** كل عملية تكتب جدول الدومين أولًا، ثم Audit append-only.

**PR المستهدف:** PR-02 ثم PRs الوحدات.

### GAP-P0-04 — Idempotency غير عام

**الحالة:** مكتمل نسبيًا للتقرير اليومي فقط.

**الخطر:** تكرار Webhook، الضغط المزدوج، Retry بعد انتهاء مهلة، أو إعادة رفع ملف.

**الإغلاق المطلوب:**

- `idempotency_key` إلزامي.
- Unique index في قاعدة البيانات.
- إعادة النتيجة الأصلية عند التكرار.
- عدم الاعتماد على فحص ثم Insert منفصلين.

**PR المستهدف:** PR-02.

### GAP-P0-05 — الذرية غير مفروضة على كل العمليات

**الحالة:** مفتوح.

**الدليل:** بعض العمليات تنفذ Patch/Insert/Send في خطوات منفصلة؛ الحذف التجريبي يستخدم `Promise.allSettled`.

**الخطر:** نجاح جزئي بين السجل الأصلي والاعتماد والتدقيق والإشعار.

**الإغلاق المطلوب:** RPC/Transaction لكل عملية حساسة، مع Outbox داخل نفس المعاملة.

**PR المستهدف:** PR-02 وPR-03.

### GAP-P0-06 — صلاحيات متفاوتة بين الدور والـCapability وAdmin

**الحالة:** مفتوح.

**الدليل:** `requireCapability` موجود، لكن مسارات تستخدم `requireAdmin` وفحوص أدوار مباشرة داخل Bot files.

**الخطر:** منح صلاحيات أوسع من الوظيفة وعدم اتساق المنع بين الموقع وتيليجرام.

**الإغلاق المطلوب:** كل فعل يحمل Capability محددة، مع فصل الإنشاء والاعتماد والترحيل والعكس.

**PR المستهدف:** PR-09، مع تمهيد في PR-02.

### GAP-P0-07 — الإشعارات لا تمر كلها عبر Outbox

**الحالة:** مفتوح.

**الدليل:** `notification_outbox` موجود، لكن إرسالًا مباشرًا متكررًا عبر `sendMessage()` ما زال قائمًا.

**الخطر:** لا توجد إعادة محاولة موحدة، وقد يصعب إثبات التسليم والفشل.

**الإغلاق المطلوب:** Outbox إلزامي، حالات `pending/sent/failed/retrying/dead_letter`، Worker idempotent.

**PR المستهدف:** PR-02.

### GAP-P0-08 — اختبارات القبول الواقعية غير مكتملة

**الحالة:** مفتوح وفق Issue #41.

**المطلوب:** اختبارات الموقع وتيليجرام والصوت وExcel والطباعة والصلاحيات والأعطال على بيانات اختبار معزولة ثم أدلة قبول.

**PR المستهدف:** PR-12.

## P1 — فجوات تشغيلية ورقابية

### GAP-P1-01 — حالات العمليات غير موحدة

مجموعات حالات مختلفة بين المبيعات والعمليات والاستيراد. يلزم State Machine لكل نوع مع انتقالات صريحة ومنع تعديل `posted`.

### GAP-P1-02 — مزامنة Legacy تحدث أثناء القراءة

`syncLegacyOrders()` قد يكتب في `sales_orders` عند عرض القائمة. يلزم Backfill مستقل، تقرير مقارنة، ثم إزالة الكتابة من مسار القراءة.

### GAP-P1-03 — الموردون والمشتريات غير مكتملين من البداية للنهاية

لا يوجد إثبات موحد لدورة: طلب شراء → عروض → مقارنة → اعتماد → أمر شراء → استلام → فاتورة مورد → استحقاق → سداد.

### GAP-P1-04 — المخزون غير مكتمل رقابيًا

يلزم إثبات الاستلام والصرف والتحويل والجرد والتسوية وحد إعادة الطلب ومنع الرصيد السالب.

### GAP-P1-05 — الإنتاج غير مربوط بالكامل بأوامر البيع والتكلفة

تقارير الخرسانة والبلوك موجودة، لكن يجب إثبات الربط بالمواد والهالك والتكلفة والأمر والربحية.

### GAP-P1-06 — الديزل لا يملك قبولًا نهائيًا لكل حركة

يجب منع الحركة غير المرتبطة بأصل، وتوحيد اللوحات، ووضع الاستثناءات في إعدادات لا في أسماء ثابتة داخل الكود.

### GAP-P1-07 — الصيانة غير مثبتة كدورة كاملة

بلاغ → فحص → تشخيص → عرض → اعتماد → إصلاح → اختبار → إغلاق، مع منع الإغلاق دون التكلفة والمستندات عند انطباق الشرط.

### GAP-P1-08 — الحضور يحتاج اختبار منع الانتحال

يلزم ربط الموظف والحساب والجهاز والموقع، ومنع التسجيل نيابة عن مستخدم آخر، وتوثيق استثناءات GPS.

### GAP-P1-09 — العهدة تحتاج دورة كاملة

طلب وإصدار واستلام وإرجاع وفقد/تلف وتسوية وإثبات إلكتروني.

### GAP-P1-10 — الصوت والصور تحتاج مسودة إلزامية

لا تنفذ حركة حساسة من التفريغ أو الاستخراج مباشرة. يجب عرض النص والحقول والثقة ثم تأكيد/تعديل/إلغاء.

### GAP-P1-11 — التقارير النهائية غير موحدة التحقق

كل تقرير نهائي يحتاج رقمًا مرجعيًا، نسخة النظام، بصمة، QR، وسجلًا في `document_registry` أو سجل موحد مماثل.

### GAP-P1-12 — معالجة الأخطاء قد تخفي التعطل

`safeSelect()` وقوالب `.catch(()=>[])` قد تعرض صفر بيانات بدل خطأ مصدر. يجب تصنيف الخطأ وإظهاره لمركز الرقابة.

## P2 — دين فني

### GAP-P2-01 — `legacy.html` ضخم

الاختبار يمنع نموه لكنه لا يعالج صعوبة الاختبار والصيانة. يجب استخراج الوحدات تدريجيًا دون إعادة كتابة جماعية.

### GAP-P2-02 — `management.js` متعدد المسؤوليات

ينبغي تقسيمه إلى خدمات: conversations, operations, approvals, notifications, reports, documents.

### GAP-P2-03 — `bot-sales.js` يجمع Adapter وDomain وPresentation

ينبغي إبقاء parsing والجلسات والتنسيق في Adapter، ونقل الحفظ والحالات والاعتماد إلى Sales Service.

### GAP-P2-04 — عقود DTO غير مركزية

يلزم Schemas مشتركة للعميل وأمر البيع والفاتورة والتحصيل والملف مع تطبيع موحد للأرقام العربية والإنجليزية.
