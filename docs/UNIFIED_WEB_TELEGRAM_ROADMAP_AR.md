# الخطة الرئيسية لتوحيد الموقع وتيليجرام

## 1. الهدف النهائي

تحويل نظام مصنع بن حامد إلى منصة تشغيل ورقابة موحدة لها واجهتان:

- الموقع.
- بوت تيليجرام.

الواجهتان لا تملكان منطقًا ماليًا أو تشغيليًا منفصلًا. كل عملية تمر عبر نفس خدمات النظام، ونفس الصلاحيات، ونفس الاعتماد، ونفس قاعدة البيانات، ونفس سجل التدقيق.

**معيار النتيجة:** أي عملية تسجل من تيليجرام تظهر فورًا في الموقع، وأي عملية تسجل أو تعتمد من الموقع يمكن متابعتها من تيليجرام، دون ازدواج أو فقد أو اختلاف في النتائج.

## 2. الحالة الحالية

المشروع يحتوي على مركز رقابة، صلاحيات، نسخ احتياطية، اختبارات استعادة، محاسبة، تقارير يومية، وتكامل واسع مع تيليجرام. تبقى Issue #41 مفتوحة لأن الجاهزية النهائية تحتاج:

- إدخال البيانات الحقيقية.
- إغلاق فجوة الفاتورة والقيد من أمر البيع.
- توحيد العمليات التي ما زالت تعتمد على `audit_log`.
- اختبارات قبول واقعية من الموقع وتيليجرام والصوت وExcel والطباعة.
- توقيعات التسليم.

PR #77 أصلح حفظ أوامر البيع والعملاء والموظفين القادمين من تيليجرام داخل الجداول الفعلية. لكنه لم يحول تغيير حالة أمر البيع إلى فاتورة وقيد محاسبي.

## 3. قواعد التنفيذ الملزمة

1. لا تعديل مباشر على `main`.
2. كل مرحلة في فرع وPR مستقلين.
3. لا Migration دون Backup مشفر ناجح وفحصه، وRestore Drill معزول عند التغيير الجوهري.
4. لا حذف بيانات إنتاج.
5. لا إعادة هيكلة واسعة داخل ملف ضخم دون اختبارات رجوع.
6. لا منطق أعمال داخل HTML أو Telegram callbacks.
7. لا أسرار في الكود أو Issues أو السجلات.
8. لا إعلان جاهزية قبل اختبارات القبول الواقعية.
9. كل PR ينجح في الاختبارات وSyntax وAPI contracts وFunction count وVercel build وفحص الأسرار.
10. لا يجمع PR واحد أكثر من مرحلة تشغيلية كبيرة.

```text

  documents/
```

## 5. عقد العملية الموحد

كل عملية تتضمن:

- `operation_id`.
- `operation_type`.
- `source`: `web` أو `telegram`.
- `source_reference`.
- المستخدم والدور والصلاحية.
- تاريخ الإنشاء والحالة.
- البيانات قبل وبعد.
- صاحب الاعتماد وتاريخه.
- مرجع القيد المحاسبي.
- مرجع الملف أو المستند.
- `idempotency_key`.
- سجل الخطأ وإعادة المحاولة.

## 6. دورة الحياة

```text
draft
  -> pending_review
  -> approved
  -> posted
  -> completed
```

الحالات الاستثنائية:

- `rejected`.
- `cancelled`.
- `reversed`.
- `failed`.
- `retry_pending`.

لا تعدل عملية `posted` مباشرة؛ التصحيح يتم بعكس أو مستند تعديل موثق.

## 7. منع التكرار

يبنى `idempotency_key` من:

- نوع العملية.
- المستخدم.
- المصدر.
- رقم رسالة تيليجرام أو بصمة الملف.
- التاريخ المرجعي.
- البيانات الأساسية.

يجب أن يمنع:

- إعادة إرسال Excel.
- ضغط الاعتماد مرتين.
- تكرار Webhook.
- Refresh بعد الحفظ.
- Retry بعد انقطاع الاتصال.

## 10. PR-03 — العملاء والمبيعات والفواتير والتحصيل

### العملاء

- إنشاء وتعديل وإيقاف وتنشيط.
- حد ائتمان وأيام سداد.
- كشف حساب وأعمار دين.
- آخر المبيعات والتحصيلات.
- تطبيع الأرقام العربية والإنجليزية.

### عروض الأسعار

- طلب عرض.
- عرض فعلي بأصناف وكميات وأسعار.
- اعتماد وتحويل لأمر بيع.
- PDF وQR ورقم تحقق.
- إرسال نسخة إلى تيليجرام.

### أوامر البيع

فصل الخرسانة والبلوك، مع الحالات التشغيلية اللازمة وربطها بدورة العملية العامة.

### الفواتير والقيود

- `invoiced` ينتج فاتورة فعلية.
- اعتماد الفاتورة ينتج قيدًا واحدًا.
- مرجع الفاتورة وأمر البيع داخل القيد.
- منع القيد المكرر.
- الإلغاء بعد الترحيل بعكس رسمي.
- منع `audit_log` من العمل كبديل للفاتورة أو القيد.

### التحصيل

- نقدي ونقاط بيع وتحويل.
- خزينة صحيحة.
- FIFO أو توزيع يدوي موثق.
- كشف غير الموزع.
- منع سند القبض المكرر.

- عرض الفروقات قبل الاعتماد.

- ترحيل ذري.
- إعادة النتيجة إلى مصدر الملف.
- سجل للأسطر المرفوضة والتحذيرات.

## 13. PR-06 — الأصول والديزل والصيانة

### سجل الأصل

رقم الأصل، اللوحة، النوع، السائق، مركز التكلفة، العداد، الديزل، الصيانة، المستندات، الحالة.

### الديزل

- إدخال من الموقع أو تيليجرام.
- مطابقة اللوحة والأصل والعداد والكمية والتكلفة.
- محطة/خزان وانحراف استهلاك.
- منع السجل غير المرتبط.
- الاستثناءات من الإعدادات لا من أسماء ثابتة.

### الصيانة

```text
بلاغ -> فحص -> تشخيص -> عرض سعر -> اعتماد -> إصلاح -> اختبار -> إغلاق
```

لا إغلاق دون التكلفة والفني/المورد ووصف العمل وقطع الغيار والمستندات ونتيجة الاختبار.

## 15. PR-08 — الصوت والصور والملفات

### الصوت

```text
Receive -> Store -> Transcribe -> Detect Intent -> Extract
-> Draft -> User Confirm/Edit/Cancel -> Execute
```

لا تنفذ حركة حساسة مباشرة من الصوت.

### الصور

- قطعة غيار: رقم وماركة ومقاس.
- فاتورة: مسودة فقط.
- مستند موظف: تصنيف وربط بعد التأكيد.
- إظهار سبب الفشل والبيانات المقروءة.

### الملفات

كل ملف يملك بصمة ومالكًا ونوعًا وكيانًا مرتبطًا وتاريخ رفع وحالة فحص وصلاحية وصول.

## 16. PR-09 — الصلاحيات وسجل التدقيق

صلاحيات حسب الفعل، مثل:

- `sales.order.approve`.
- `sales.invoice.post`.
- `collections.allocate`.
- `accounting.post`.
- `inventory.adjust`.
- `maintenance.close`.
- `permissions.manage`.
- `backup.verify`.

الضوابط:

- فصل الخرسانة عن البلوك.
- منع الاعتماد الذاتي للحساس.
- تسجيل منح وسحب الصلاحيات.
- إنهاء الجلسات عند الإيقاف.
- ربط الموقع وتيليجرام بهوية موظف واحدة.
- الأدوات العليا للمالك أو التفويض الصريح.

Audit غير قابل للتعديل من الواجهة، ويسجل المنفذ والمصدر والوقت والقبل/بعد والنتيجة والاعتماد ومراجع Telegram/IP/device/file عند توفرها.

## 20. تعريف الجاهزية النهائية

لا يعتبر النظام جاهزًا إلا عند تحقق الآتي:

- كل عملية أساسية تعمل من الموقع وتيليجرام بنفس الخدمة.
- لا عمليات محفوظة في `audit_log` فقط.
- الفواتير تنتج قيودًا صحيحة مرة واحدة.
- لا تكرار عند إعادة الطلب.
- لا تحصيلات غير موزعة بلا مبرر.
- لا ديزل بلا أصل.
- لا صلاحيات زائدة.
- كل تعديل حساس له منفذ ومراجع ومعتمد.
- Backup حديث ومشفر ومتحقق منه.
- Restore Drill ناجح.
- اختبارات الموقع وتيليجرام وExcel والطباعة ناجحة.
- تقارير المدير قابلة للطباعة والتحقق.
- البيانات الحقيقية مكتملة.
- توقيع الإدارة والمالية والتشغيل ومدير النظام.
- `Production Readiness = Success`.
