# تقييم OpenAuth لمصنع بن حامد

## النتيجة التنفيذية

لا يُدمج OpenAuth داخل النسخة التشغيلية الحالية الآن. يُحتفظ به كخيار لمرحلة لاحقة عندما يصبح للمصنع أكثر من تطبيق ويب أو تطبيق جوال يحتاج تسجيل دخول مركزيًا.

## ما يقدمه المستودع

- مزود هوية مركزي قابل للاستضافة الذاتية.
- بروتوكول OAuth 2.0 وإصدار Access Token وRefresh Token.
- تشغيل على Node.js وBun وAWS Lambda وCloudflare Workers.
- مزودات دخول مثل GitHub وكلمة المرور، مع إمكانية إنشاء مزود مخصص.
- واجهة دخول قابلة للتخصيص.
- ترخيص MIT.
- الحزمة المراجعة `@openauthjs/openauth` إصدار 0.4.3، والمشروع يعلن أنه Beta.

## الحدود المهمة

OpenAuth لا يدير سجل المستخدمين أو الأدوار أو الموظفين. بعد نجاح تسجيل الدخول يستدعي Callback، ويجب على برنامجنا البحث عن المستخدم أو إنشائه داخل `app_users` وتطبيق الصلاحيات بنفسه.

كما يحتاج مخزن KV مستقلًا لرموز التحديث وكلمات المرور والبيانات الأمنية المحدودة. هذا يعني إضافة خدمة هوية ومخزن جلسات ومسار نشر وتشغيل جديد.

## مقارنة بالوضع الحالي

النظام الحالي لديه بالفعل:

- `app_users` والأدوار والصلاحيات الدقيقة.
- ربط Telegram بالمستخدم والموظف.
- جلسة جهاز آمنة داخل الموقع عبر Cookie موقعة من الخادم.
- فصل صلاحيات الاعتماد والتكاليف والإدارة.
- قيود Vercel على عدد عمليات النشر والوظائف.

إضافة OpenAuth الآن لن تحسن تقارير العملاء أو مساعد المنتجات، وستزيد نقاط الفشل وإدارة الأسرار والجلسات.

## الاستخدام الصحيح مستقبلًا

يصبح OpenAuth مفيدًا عند وجود اثنين أو أكثر من الآتي:

1. موقع الإدارة وتطبيق جوال للسائقين وبوابة موظفين منفصلة.
2. رغبة في تسجيل دخول موحد بين تطبيقات متعددة.
3. دخول بالبريد وكلمة المرور أو Google أو Microsoft بدل جلسة الجهاز فقط.
4. أطراف خارجية تحتاج OAuth رسميًا للوصول إلى API محددة.

## تصميم التكامل المقترح

- نشر OpenAuth كخدمة مستقلة مثل `auth.binhamid.example`، وليس داخل Router الإنتاج الحالي.
- تعريف Subject بسيط يحتوي `appUserId` و`sessionVersion` فقط.
- عدم وضع الراتب أو الحد الائتماني أو جميع الصلاحيات داخل JWT طويل العمر.
- قراءة الدور والصلاحيات الحالية من `app_users` و`role_capabilities` في كل عملية حساسة.
- ربط هوية OpenAuth بسجل `app_users` دون إنشاء سجل مستخدم موازٍ.
- إبقاء Telegram قناة هوية مستقلة مرتبطة بالمستخدم نفسه.
- إلغاء الرموز والجلسات عند تعطيل الموظف أو تغيير دوره.
- تنفيذ تجربة على بيئة Staging قبل أي تحويل للإنتاج.

## القرار

الاستفادة الحالية من المستودع معمارية وليست نسخ الكود: اعتماد OAuth القياسي عند تعدد التطبيقات، وفصل مزود الهوية عن بيانات المستخدم والصلاحيات. التنفيذ مؤجل حتى توجد حاجة تشغيلية حقيقية لتسجيل دخول مركزي متعدد التطبيقات.
